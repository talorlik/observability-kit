# Configuration Guide

installation: the unified configuration document, how changes
propagate to the wrapped systems, how drift is detected, and how
configuration is rolled back. The operator runbook form of this

## One Document, One Pipeline

All persistent platform configuration flows from a single
schema-validated JSON document through one pipeline: render, commit,
reconcile. There is no shortcut stage. A change that skips render is
a hand edit, a change that skips commit is untracked state, and a
change that skips reconcile is a direct API write; all three are
contract violations, and the first two are exactly what drift
detection exists to catch.

The renderer runtime is `obskit render`, `obskit drift`, and
`obskit rollback`, decided by
[ADR-0003](../adr/ADR_0003_CONFIG_RENDERER_ARCHITECTURE.md) and
governed by
[RENDERER_ARCHITECTURE_CONTRACT_V1.yaml](../../contracts/management/RENDERER_ARCHITECTURE_CONTRACT_V1.yaml)
and
[PROPAGATION_RECONCILIATION_CONTRACT_V1.yaml](../../contracts/management/PROPAGATION_RECONCILIATION_CONTRACT_V1.yaml).

## The Unified Configuration Document

The document validates against
[UNIFIED_CONFIG_SCHEMA_V1.json](../../contracts/management/UNIFIED_CONFIG_SCHEMA_V1.json).
Its `config` block covers eight areas: `retention`, `tracing`, `tls`,
`alerting`, `dashboards`, `graph`, `tenancy`, and `delivery`. Its
`bindings` block maps each configuration key to the wrapped system
and file path (the binding's `render_target`) that the renderer
writes.

The document is the source of truth. Native configuration files,
Helm values, provisioning files, saved objects, and alert rules are
renderer outputs; editing them by hand defeats drift detection and is
detected as drift even inside Git.

## The Propagation Pipeline

Every configuration change travels three stages in order:

1. Render. The renderer reads the schema-validated document and its
   propagation bindings and produces native configuration files at
   each binding's `render_target`. Rendering is the only way native
   config under a registered configuration surface changes.
2. Commit. Rendered outputs land as one Git commit per document
   change on the delivery branch the GitOps controller watches. The
   commit is the unit of propagation, review, audit, and rollback,
   and it carries two required trailers,
   `Unified-Config-Schema-Version` and
   `Unified-Config-Document-Digest`, binding it back to the document
   revision that produced it.
3. Reconcile. Argo CD applies the committed rendered state to the
   cluster and the wrapped systems. The platform never bypasses the
   controller.

Drift detection runs continuously against the last verified rendered
commit as a fourth, standing stage.

Rendering is deterministic and idempotent by contract: the same
document bytes produce byte-identical rendered bytes (no timestamps,
hostnames, random identifiers, or ordering drift), and an unchanged
document produces no diff and therefore no commit. Every rendered
file carries the generated-file header marker
`GENERATED by the unified configuration renderer - DO NOT EDIT BY
HAND`; formats that cannot carry comments carry the marker in a
sibling manifest listing the rendered artifacts and their digests.

## Rendering Configuration

```bash
.venv/bin/obskit render \
  --document ./unified-config.json \
  --repo-root . \
  --commit-message-out ./commit-message.txt
```

The command validates the document against the schema, renders every
binding through its cataloged strategy, writes the render manifest
(by default `gitops/UNIFIED_CONFIG_RENDER_MANIFEST.json` under the
repository root, overridable with `--manifest-out`), and prepares the
propagation commit message with the required trailers. You review the
diff and commit; the renderer writes files only.

Use `--check` to plan without writing: it exits `3` listing the
targets that would change and `0` when the tree already matches the
document. This is the pre-commit gate for configuration pull
requests.

| Flag | Meaning |
| ---- | ---- |
| `--document <path>` | Required. Unified configuration document as JSON. |
| `--contracts-dir <path>` | Repository contracts directory (default: `./contracts`). |
| `--repo-root <path>` | Repository root render targets are written under (default: `.`). |
| `--manifest-out <path>` | Override path for the render manifest. |
| `--commit-message-out <path>` | Write the prepared propagation commit message to this path. |
| `--check` | Plan only: exit `3` when targets would change, `0` when the tree matches. |

## Render Strategies

Every `(unified_key, system)` binding pair maps to exactly one
strategy in the renderer's strategy catalog; a binding without a
cataloged strategy fails the render before any write. The four
strategies are:

- `json-path-patch` patches a value at a JSON path inside an existing
  JSON artifact.
- `yaml-line-patch` patches a value inside an existing YAML artifact
  while preserving the surrounding content.
- `owned-artifact` regenerates a whole file the renderer owns
  end-to-end.
- `presence-gated` renders content whose presence depends on a
  document condition (for example, a feature that is off).

The full catalog with each binding's strategy lives in
[RENDERER_ARCHITECTURE_CONTRACT_V1.yaml](../../contracts/management/RENDERER_ARCHITECTURE_CONTRACT_V1.yaml).

## Detecting Drift

Live state that diverges from the rendered commit is drift, no matter
how it got there: a hand edit to a rendered file, an out-of-band
commit, or a change applied directly to a wrapped system. The diff
surface is:

```bash
.venv/bin/obskit drift \
  --document ./unified-config.json \
  --repo-root .
```

The command is read-only. It compares the expected rendered bytes for
the document against the target tree and exits `0` when clean and `3`
when drift is detected, printing the rendered-versus-live diff report
to stdout (add `--report-out` to also write it to a file inside the
repository root). This same surface feeds the platform's
`config-drift-detected-per-system` alert path, so an on-call operator
and a CI job see identical drift reports.

## Responding to Drift

Drift has exactly two legitimate resolutions, because the document is
the source of truth:

- The change was wanted. Make it in the unified document, re-render,
  and commit. The rendered tree then matches intent again.
- The change was not wanted. Re-render from the current document (or
  revert the offending commit) so the rendered state is restored.

Never resolve drift by editing rendered files to match live state;
that converts detected drift into silent drift. The
[unified configuration runbook](../runbooks/UNIFIED_CONFIGURATION_RUNBOOK.md)
has the full drift response procedure, including how the drift alert
is routed.

## Rolling Back Configuration

Rollback is a path back through the same pipeline, never a separate
apply channel: `obskit rollback` re-renders a prior document revision
through the identical render-and-commit flow.

```bash
.venv/bin/obskit rollback \
  --document ./unified-config-previous.json \
  --expected-manifest ./previous-render-manifest.json \
  --mode dry-run
```

The command is dry-run by default: it plans the rollback re-render
and reports without writing anything. `--mode real` executes the
re-render and re-verifies the tree afterwards. When
`--expected-manifest` is given (the render manifest previously
committed for that document revision), the planned manifest must be
byte-identical to it or the rollback refuses to proceed in either
mode; this is the deterministic rollback proof that you are getting
back exactly the state that revision produced.

The rollback drill, `scripts/ops/run_config_rollback_drill.sh`,
exercises this path end to end and is also dry-run by default. Run it
on a schedule so rollback stays a rehearsed operation, not a
first-time event during an incident.

## Editing Through the Portal

The management portal edits the same unified document and delegates
to the same renderer, GitOps-only. A portal edit becomes a rendered
commit exactly as a CLI edit does; anything applied outside that path
is drift by construction. See the
[management portal guide](../runbooks/MANAGEMENT_PORTAL_GUIDE.md) for
the portal workflow.
